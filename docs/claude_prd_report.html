一、PRD 目标与问题定义

这份 PRD 的核心命题不是“做一个活动来了以后临时筛 SKU 的工具”，而是把商品运营的临时救火，转成日常、可追溯、可协同的执行系统。

核心口号
让每个 SKU 平时就处于 Ready 状态
产品定位
企业 Agent 体系中的运营执行层
能力核心
自动采集、自动诊断、自动模拟、人工 Review
安全原则
LLM 负责理解，规则引擎负责判断

PRD 解决的真实问题

  • 多平台商品数据分散在电商后台、ERP、报表、活动系统与售后系统中。
  • 活动规则复杂，且往往在活动发布后才临时拉数、筛数、协同确认。
  • 库存不足、证书缺失、好评率下降、互斥活动、价格口径冲突等问题，常在最后一刻暴露。
  • 如果直接把业务决策交给 LLM，会带来幻觉、误判与不可追溯风险。

PRD 的正确方向

  • 把“活动临时报名判断”升级成“长期 SKU 健康管理”。
  • 把“Excel 手工筛选”升级成“规则解析 + 准入模拟 + Review Gate”。
  • 把“聊天问答”降级为自然语言控制台，而不是业务核心本体。
  • 把 A 战略 Agent、D 风控 Agent 的信号，落到 SKU 级执行建议上。
一句话总结： PRD 的产品叙事是成立的，且适合黑客松路演；真正要收紧的是详细设计边界，而不是推翻方向。

四、具体价值点

这个方案的价值，不在“AI 帮你回答一个问题”，而在于把多源数据、规则约束和执行闭环统一起来。

1. 把临时救火变成持续运营

传统流程是活动来了才拉表筛数；SKU Ready Agent 把商品健康前移，平时就识别库存、证书、信息完整度、经营风险等问题。

2. 把规则理解与业务判断分开

LLM 负责把自然语言活动规则解析成结构化 DSL，真正的准入判断交给规则引擎执行，减少幻觉风险。

3. 把不可追溯建议变成证据驱动决策

每个结论都能指向字段、来源、采集时间与规则证据，适合运营、商品、供应链、法务共同 Review。

4. 把聊天框降级为控制台

Dashboard、详情页、Review 工作台和报告都可直接查询服务端；Chat 只是自然语言入口，不成为系统单点。

5. 天然适配多 Agent 体系

战略 Agent 给市场机会信号，风控 Agent 给边界信号，SKU Ready Agent 把它们翻译成 SKU 级可执行建议。

6. 黑客松上可讲清“AI 安全”

Fail-closed、Review Gate、Zod 校验、Evidence Chain 这些设计，能让评委相信这不是“把大模型硬套进业务”。

五、核心注意点与设计边界

这些不是“以后再优化”的细节，而是决定这套实现会不会从 Demo 代码快速失控的关键边界。

注意点 建议做法 风险
长期健康 vs 活动准入 拆成 healthStatus 与 eligibilityStatus 不拆会把活动门槛污染为长期标准
规则 DSL 最少支持 threshold / field_compare / boolean_block / mutual_exclusion / category_quota / manual_review / data_required DSL 不足会让模拟器退化成硬编码 demo
SKU identity 先定义 platform + storeId + skuId 与 canonicalSkuId 的关系 档案页、跨平台汇总、Chat 查询会没有稳定主语
证据链 定义 EvidenceRef：sourceType / sourceId / field / rawValue / normalizedValue / collectedAt / ruleId / evidenceText Review Gate 缺乏真正的追溯能力
时态 / 生效时间 明确 latest snapshot、latest valid diagnosis、simulation validity 过期数据、旧规则结果会污染 Dashboard 和 API
类目上限等集合规则 区分 item-level rules 与 set-level rules 无法靠单 SKU 判断解决
P0 范围控制 先跑通主闭环，Chat 和复杂工作流只做薄层 黑客松现场最容易卡在 UI 集成和非主链路
建议的明天前必须钉死的 5 个边界： 稳定长期主语、SKU 身份规则、Rule DSL、健康状态与活动状态拆分、结构化 EvidenceRef。

六、建议的 MVP 主链路

如果目标是“明天可以稳定演示”，建议把所有需求收敛成一条能完整跑通、可解释、可回放的主链路。

主闭环

  1. 模拟商品后台页面提供可识别表格。
  2. 插件扫描当前页面并做字段预览。
  3. /api/ingest 做 Zod 校验、标准化、入库 snapshot。
  4. 生成 deterministic health diagnosis。
  5. 用户粘贴活动规则，LLM 解析为 Rule DSL。
  6. 规则引擎执行活动准入模拟。
  7. 生成 Review 清单与报告摘要。

可降级项

  1. Chat 可以只做薄工具调用，不做复杂多轮智能体。
  2. WorkflowRun / WorkflowStep 只做轻量日志，不做通用编排引擎。
  3. 外部 Agent 事件先做静态模拟卡片。
  4. CSV 导入、跨平台同 SKU 汇总、复杂权限系统后置。
  5. 报告先输出 Markdown 或 HTML 片段即可，不必先做 PDF 导出。

十三、最终建议

如果目标是“明天黑客松稳”，建议采用“产品叙事不动、实现边界收紧、主链路先跑通”的策略。

  1. 保留 PRD 主叙事： 继续强调这是企业 Agent 体系中的运营执行层。
  2. 先固化最小架构： SkuProfile、Snapshot、Diagnosis、RuleSet、Simulation、Review、EvidenceRef。
  3. Chat 不做核心： 先保证 API 与服务层能独立跑通，聊天只是自然语言入口。
  4. 主链路优先： 采集 → 入库 → 诊断 → 解析规则 → 模拟 → Review → 报告。
  5. 安全能力做成系统能力： 不要只停留在文案上，至少要有结构化 evidence 与 fail-closed 分支。
一句收尾： 这份 PRD 值得继续推进；它真正需要的不是“重写”，而是把 主语、规则、时态、证据、主链路 五个边界先钉死，再进入代码生成与模块初始化。
