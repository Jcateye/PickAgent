## 1. 目标

> 说明：文中出现的 `platform_auth`、`business_user`、`business_order`、`auth`、`user`、`order` 等名称均为示例，用于说明规范写法，不代表新项目默认模块划分。

本规范用于统一项目工程结构、模块边界、AI agent 工作方式与跨模块协作规则，目标是：

1. 让项目结构长期可维护
    
2. 让 AI 只在受控范围内编码
    
3. 降低上下文污染和误改风险
    
4. 让前后端边界清晰
    
5. 为后续扩展、联调、拆分留出空间
    

---

## 2. 总体架构原则

### 2.1 总原则

- 前端采用 **单仓库**
    
- 后端采用 **按模块拆分，模块分为 `business_*` 与 `platform_*` 两类**
    
- 后端核心模块允许 **独立仓库**
    
- 共享内容分为两类：
    
    - `docs`：共享规则与设计文档
        
    - `contracts`：共享契约与基础定义
        
- 默认不共享业务实现，只共享规则和契约
    
- AI 默认单模块工作，跨模块必须显式授权
    

### 2.2 设计取向

- 前端强调：统一体验、统一构建、统一状态与路由
    
- 后端强调：边界隔离、小范围编码、可独立维护
    
- 架构优先服务于 **小任务拆分 + agent 受控执行**
    
- 不为“漂亮架构”增加不必要复杂度

## 3. 仓库与目录组织

## 3.1 工作区结构

```txt
workspace/
  docs/
  contracts/
  frontend/
  platform_auth/
  business_user/
  business_order/
  platform_payment/
```

### `workspace` 的含义

- `workspace` 是本地协作根目录，也是 `frontend/` 与各后端模块的父目录
    
- `workspace` 主要用于统一组织共享文档、契约与跨模块协作
    
- `workspace` 不直接等于单一部署单元
    
- `workspace` 可以是一个总 Git 仓库，也可以只是多个独立仓库的共同父目录
    
- 日常开发、测试、发布默认在具体模块仓库内完成，不要求所有动作都在 `workspace` 根目录执行
    
- 只有跨模块联调、共享文档维护、契约协作时，才需要在 `workspace` 视角下工作

---

## 3.2 根目录职责

### `docs/`

存放共享规则、工程约束、架构说明、变更记录。

### `contracts/`

存放前后端和跨模块共享的契约定义。

### `frontend/`

前端唯一仓库。

### `business_*` / `platform_*`

后端按模块拆分后的独立仓库。

---

## 3.3 后端模块命名规则

后端模块命名只允许两类前缀，不再使用 `backend_*`：

- `business_*`：面向具体业务域的模块，例如 `business_user`、`business_order`
    
- `platform_*`：面向多个业务域复用的平台能力模块，例如 `platform_auth`、`platform_payment`
- `platform_*` 必须提供明确的跨业务域能力边界，不允许创建 `platform_common`、`platform_utils`、`platform_shared`、`platform_base` 这类泛化模块
- 若一个能力无法清晰说明其服务对象、输入输出契约与 owner，则不得归类为 `platform_*`
    
- 命名格式统一为：`前缀_领域名`
    
- 领域名使用小写英文 + 下划线，不使用中划线、不使用拼音缩写、不使用 `backend` 前缀
    
- 仓库名、目录名、包名、服务标识建议保持一致，避免同一模块出现多套叫法
    
- 如果一个模块的核心价值是承载业务流程与业务实体，归为 `business_*`
    
- 如果一个模块的核心价值是向多个业务模块提供通用能力，归为 `platform_*`

---

## 3.4 工作区与仓库运营规则

- `frontend/` 与每个后端模块都是独立工程单元，可以各自独立维护
    
- `docs/` 与 `contracts/` 是共享协作目录，不承载业务实现
    
- 构建、测试、发布、部署默认按具体模块执行，不按整个 `workspace` 一次性执行
    
- `platform_*` 模块是否存在、是否新建，属于架构设计阶段决策
    
- 普通开发任务默认只在既有模块内完成，不得因局部复用需求顺手新建 `platform_*`
    
- 若确认需要新增 `platform_*` 或调整模块边界，必须先记录到 `decisions.md`，再单独立项处理

# 4. docs 规范

## 4.1 docs 目录结构

```txt
docs/
  architecture.md
  engineering-rules.md
  agent-rules.md
  api-guidelines.md
  db-guidelines.md
  changelog.md
  decisions.md
```

---

## 4.2 各文档职责

### `architecture.md`

描述整体架构、仓库划分、模块边界、共享策略。

### `engineering-rules.md`

描述通用工程约束、编码规则、前后端分层规则。

### `agent-rules.md`

描述 AI agent 的默认工作边界、输入范围、输出要求、禁止事项。

### `api-guidelines.md`

描述 API 命名、请求响应结构、错误码约定、契约同步要求。

### `db-guidelines.md`

描述数据库分层原则、模型分离规则、迁移约束。

### `changelog.md`

记录重要功能完成情况。

### `decisions.md`

记录关键工程决策及原因。

---

## 4.3 文档编写原则

- `docs/` 中的内容分为两类：项目可变项 与 跨项目稳定约束
    
- 项目可变项可以因项目不同而调整，例如技术栈、框架选择、部署方式、CI 工具、监控方案
    
- 跨项目稳定约束应尽量保持统一，例如模块边界、命名规则、错误码规则、API 结构、迁移约束、agent 行为边界
    
- 若一个规则属于“所有项目都建议遵守”，应优先写入对应 guideline 文档，而不是散落在任务描述中
    
- 若一个内容只适用于当前项目，应明确标记为“项目特定决策”，避免误当作通用规则
    
---

## 4.4 各文档最小内容要求

### `architecture.md` 至少包含

- 项目目标与非目标
    
- 当前阶段采用的总体架构模式
    
- `workspace` / 仓库 / 模块划分方式
    
- `business_*` 与 `platform_*` 的边界与 owner
    
- 前后端、契约、共享目录之间的关系
    
- 当前项目确定采用的技术栈与关键基础设施
    
### `engineering-rules.md` 至少包含

- 使用语言、运行时、框架、包管理器等基础技术栈
    
- 目录分层、命名规范、文件组织方式
    
- lint / format / typecheck / test 的最低要求
    
- 依赖管理规则与禁止事项
    
- 配置管理、环境变量、日志、异常处理等通用工程规则
    
- 编码规范与代码评审要求
    
### `agent-rules.md` 至少包含

- agent 默认允许读取的范围
    
- agent 默认禁止执行的操作
    
- 单模块、跨模块、系统级任务的授权方式
    
- agent 输出格式要求
    
- 风险等级与不同风险等级下的执行要求
    
- 何时必须停止并请求人工确认
    
### `api-guidelines.md` 至少包含

- API 路径命名与资源命名规则
    
- 请求体、响应体的统一结构
    
- 成功响应与失败响应的统一格式
    
- 错误码来源、分配规则、返回方式
    
- 分页、过滤、排序、幂等、鉴权、版本兼容等通用约定
    
- OpenAPI / contracts 的同步方式
    
### `db-guidelines.md` 至少包含

- 数据 owner 规则与跨模块访问边界
    
- 表、字段、索引、约束、枚举的命名规则

- 表注释、字段注释的强制要求与书写规则
    
- 主键、审计字段、软删字段等通用字段规范
    
- 模型分层规则：存储模型、领域模型、传输模型
    
- migration 策略、回滚策略、数据修复策略
    
- 索引、唯一约束、历史数据、归档、审计等设计要求
    
### `changelog.md` 至少包含

- 变更日期
    
- 涉及模块
    
- 变更内容摘要
    
- 是否影响契约、数据、联调
    
- 必要时附迁移说明或兼容性说明
    
### `decisions.md` 至少包含

- 决策标题与唯一标识
    
- 决策背景
    
- 备选方案
    
- 最终决定
    
- 决定原因与影响范围
    
- owner、日期、状态

# 5. contracts 规范

## 5.1 contracts 目录结构

```txt
contracts/
  openapi/
  events/
  errors/
  types/
```

---

## 5.2 各目录职责

### `openapi/`

存放接口契约定义。

### `events/`

存放跨模块事件定义。

### `errors/`

存放统一错误码定义。

### `types/`

存放少量跨模块共享基础类型。

---

## 5.3 contracts 允许放的内容

- 接口 schema
    
- 事件 payload
    
- 错误码
    
- 少量公共类型
    
- 契约文档
    

---

## 5.4 contracts 禁止放的内容

- 业务 service 实现
    
- repository 实现
    
- 半业务 util
    
- 某模块私有逻辑
    
- 私有 helper
    

---

## 5.5 contracts owner 规则

- 每一份契约都必须有明确 owner 模块
    
- owner 模块负责该契约的语义定义、兼容性、版本演进与变更说明
    
- 非 owner 模块可以提变更需求，但不得未经确认直接改变契约语义
    
- owner 信息建议记录在契约文件同级 README、目录说明或文件头注释中
    
- 一个契约只能有一个主 owner，避免多方共同拥有导致失控
    
---

## 5.6 contracts 兼容与变更规则

- 优先做向后兼容变更，不要轻易做 breaking change
    
- breaking change 必须通过新版本文件、版本目录或明确兼容窗口处理，不得静默覆盖旧语义
    
- 已被其他模块使用的契约，默认遵循追加优先原则：新增字段、新增错误码、新增事件版本，优先于直接删除或重命名
    
- 错误码一经发布，不得复用旧编号表达新语义
    
- 事件结构一经发布，不得在原版本中直接删除关键字段
    
- OpenAPI 结构若需破坏性调整，必须先说明受影响模块与迁移步骤
    
---

## 5.7 contracts 发布顺序

- 契约变更任务必须明确标注 owner 模块、受影响模块、兼容策略
    
- 若变更影响 producer / consumer，默认顺序是：先完成契约评审，再由 producer 落地兼容实现，最后推动 consumer 迁移
    
- 在全部 consumer 完成迁移前，不得删除兼容层
    
- 契约变更应尽量与模块发布记录同步写入 `changelog.md` 或相关变更文档
## 5.8 contracts 废弃规则  
  
- 任何准备废弃的 API、事件、错误码、类型，必须先标记 deprecated，不得直接删除  
- 必须标记废弃时间、owner、替代方案、预计删除时间  
- 在所有 consumer 完成迁移前，不得移除兼容层
---
# 6. 前端工程规范

## 6.1 前端原则

- 前端保持单仓库
- 前端按业务模块拆目录
- 页面、模块、状态、接口、类型尽量在模块内闭环
- 公共能力统一收敛，但要克制
- 前端 `src/modules/*` 的命名应尽量与后端对应领域保持一致

---

## 6.2 前端目录结构

```txt
frontend/
  src/
    app/
      bootstrap/
      router/
      providers/
      guards/
      layouts/
      store/
      styles/

    modules/
      auth/
        pages/
        components/
        api/
        schemas/
        hooks/
        store/
        types/
        utils/

      user/
        pages/
        components/
        api/
        schemas/
        hooks/
        store/
        types/
        utils/

      order/
        pages/
        components/
        api/
        schemas/
        hooks/
        store/
        types/
        utils/

    shared/
      ui/
      api/
      config/
      hooks/
      lib/
      types/
      utils/

  public/
  package.json
  README.md
```

---

## 6.3 前端分层规则

### `app/`

放应用级能力：

- 路由
    
- provider

- guards

- layouts
    
- 应用入口
    
- 全局 store 初始化
    
- 全局样式
    

### `modules/*/pages`

放页面级组件。

### `modules/*/components`

放模块内部复用组件。

### `modules/*/api`

放当前模块的 API 请求封装。

### `modules/*/schemas`

放当前模块的表单、查询、响应校验规则。

### `modules/*/store`

放当前模块状态管理。

### `modules/*/types`

放当前模块类型定义。

### `modules/*/hooks`

放当前模块专用 hooks。

### `shared/`

放无明显业务语义的通用能力。

### `shared/ui`

放真正通用 UI 组件。

---

## 6.4 前端共享边界规则

可以放入全局共享的例子：

- Button
    
- Modal
    
- Table
    
- EmptyState
    
- useDebounce
    
- request client
    
- formatDate
    

不应放入全局共享的例子：

- OrderStatusTag
    
- MerchantSelector
    
- AuthLoginForm
    
- RecyclePriceCard
    

原则：

**带明显业务词的代码，优先留在模块内部。**

---
## 6.5 前端路由与权限规则  
  
- 全局路由注册放在 `app/router`  
- 模块页面路由定义可由模块提供，再统一汇总到应用路由层  
- 权限判断策略统一收敛在应用层或明确的权限模块中，不得分散写在各页面组件内部  
- 页面级临时判断允许存在，但不得替代统一权限机制
---
## 6.6 前端禁止事项

- 不得把所有 API 堆到全局 `api/`
    
- 不得把所有类型堆到全局 `types/`
    
- 不得为了复用过早抽公共组件
    
- 不得为“优雅”重构全局页面结构
    
- 不得默认改动其他模块页面
    

---

# 7. 后端工程规范

## 7.1 后端原则

- 后端模块按 `business_*` 与 `platform_*` 两类命名
    
- 核心模块可独立仓库
    
- 模块内部按职责分层
    
- 共享规则与契约，不共享业务实现
    
- 允许未来演进为独立服务，但当前不强制真实微服务化
    

---

## 7.2 后端模块分类建议

常见分类参考：

- `business_*`：user、order、merchant、inventory、crm
    
- `platform_*`：auth、payment、message、file、search、workflow
    
判断优先级：

- 优先看模块服务对象，是单一业务域还是多个业务域
    
- 再看模块复用方式，是业务闭环能力还是平台通用能力
    
- 仅因技术实现复杂，不会自动变成 `platform_*`
    
- 模块分类与是否新建 `platform_*`，应在架构设计阶段确定，不在普通开发任务中临时决定
    
---

## 7.3 后端仓库目录结构

以 `platform_auth/` 为例：

```txt
platform_auth/
  src/
    api/
      controllers/
      dto/
      routes/

    application/
      services/
      use-cases/

    domain/
      entities/
      value-objects/
      rules/
      types/

    infrastructure/
      repository/
      persistence/
      external/
      mappers/

    shared/
      errors/
      utils/
      constants/

    bootstrap.ts
    index.ts

  tests/
    unit/
    integration/

  scripts/
  package.json
  README.md
```

---

## 7.4 后端分层职责

### `api/`

负责：

- HTTP 路由
    
- 请求与响应结构
    
- 参数校验
    
- DTO
    

禁止：

- 直接查数据库
    
- 直接写业务规则
    
- 直接跨模块调用内部实现
    

### `application/`

负责：

- 业务流程编排
    
- use case
    
- 事务边界
    
- 调用 domain / repository / external
    

禁止：

- HTTP 细节
    
- 表结构定义
    
- 无边界的大而全工具函数
    

### `domain/`

负责：

- 核心业务规则
    
- 实体
    
- 值对象
    
- 状态流转
    
- 领域类型
    

禁止：

- ORM 实现
    
- 三方网络调用
    
- 接口出参拼装
    

### `infrastructure/`

负责：

- 数据库访问
    
- cache
    
- queue
    
- 第三方集成
    
- mapper
    

禁止：

- 决定业务规则
    
- 依赖其他模块内部私有实现
    

### `shared/`

仅用于当前仓库内部的少量共用能力，不等于全局共享仓库。
- `shared/` 只允许放当前仓库内部多处复用、且无明确业务归属的轻量能力  
- 带业务语义的代码不得因“复用”进入 `shared/`  
- 若某段逻辑只被一个 use case 使用，应留在对应 application / domain 内，而不是抽入 `shared/`

---

## 7.5 后端模块边界规则

允许：

- 依赖 `contracts`
    
- 使用公开 API / client
    
- 通过事件协作
    
- 使用统一错误码
    

禁止：

- 直接 import 别的模块 repository
    
- 直接依赖别的模块 service 内部实现
    
- 直接访问别的模块内部表结构
    
- 共享半业务逻辑
    

---

## 7.6 后端数据所有权规则

- 每个后端模块只对自己的数据结构、迁移脚本、持久化模型负责
    
- 其他模块不得直接写入该模块拥有的表、schema、collection 或索引
    
- 即使多个模块共用同一个数据库实例，也必须维持明确的数据 owner 边界
    
- 数据库迁移默认由 owner 模块执行，其他模块不得顺手修改其 migration
    
- 跨模块数据变更不得通过共享事务直接完成
    
- 若一个需求长期依赖跨模块强一致写入，优先重新审视模块边界，而不是继续增加隐式耦合
    
---

## 7.7 后端模块通信规则

- 查询类读取默认通过公开 API / client 完成
    
- 状态通知、异步解耦、跨模块广播默认通过事件完成
    
- 共享基础类型只能放在 `contracts/types` 等共享契约目录，不得反向依赖其他模块内部类型
    
- 禁止跨模块直接共享 repository、service、ORM model
    
- 禁止跨模块共享本地事务
    
- 若通信路径无法清晰归类为 API 或事件，应先补设计说明，再开始实现
    
---

# 8. 数据模型规范

## 8.1 模型必须分离

同一业务对象，至少区分以下三类：

### 1）存储模型

例如：

- `UserRecord`
    
- `OrderRecord`
    

用于数据库持久化。

### 2）领域模型

例如：

- `User`
    
- `Order`
    

用于表达业务规则与状态。

### 3）传输模型

例如：

- `CreateOrderRequestDto`
    
- `OrderDetailResponseDto`
    

用于 API 输入输出。

---

## 8.2 禁止事项

- 不得把 ORM 模型直接当接口返回对象
    
- 不得把数据库表字段语义直接暴露到全部业务层
    
- 不得用一个 model 同时承担存储、业务、传输三种职责
    

---

# 9. AI Agent 工作规范

## 9.1 默认工作模式

AI 默认只允许在 **单模块 / 单任务** 范围内工作。

---

## 9.2 默认可读取内容

- 当前模块或当前仓库代码
    
- `docs/architecture.md`
    
- `docs/engineering-rules.md`
    
- `docs/agent-rules.md`
    
- 当前任务直接相关的 `contracts`
    

---

## 9.3 默认禁止事项

- 扫描整个 workspace
    
- 修改未授权模块
    
- 顺手重构无关代码
    
- 私自新增依赖
    
- 私自升级框架或基础配置
    
- 私自修改数据库 schema
    
- 扩大任务边界
    
- 私自新建 `platform_*` 模块或调整模块边界
    

---

## 9.4 每次执行必须输出的内容

1. 任务理解
    
2. 计划修改的文件
    
3. 实现步骤
    
4. 风险点
    
5. 自测步骤
    
6. 影响范围
    

---

## 9.5 变更风险分级

### `L0` 低风险

- 文档、注释、测试、样式微调
    
- 不改变运行时行为
    
### `L1` 常规风险

- 单模块内部实现调整
    
- 不影响公开契约、不涉及迁移、不改变模块边界
    
### `L2` 中风险

- `contracts` 变更
    
- 错误码、公开 API、事件结构调整
    
- 跨模块联调任务
    
### `L3` 高风险

- 数据库 migration
    
- 认证、权限、支付、工作流等核心链路改动
    
- 依赖升级、基础配置调整、模块边界调整
    
- 新建 `platform_*` 模块
    
---

## 9.6 不同风险等级的执行要求

- `L0` 与 `L1`：在授权范围内按正常流程执行
    
- `L2`：必须额外说明兼容性影响、受影响模块、联调与验证步骤
    
- `L3`：必须额外说明回滚方案、兼容策略、验证清单，并由任务明确授权后才允许执行
    
- 若任务未标明风险等级，默认按不低于 `L1` 处理
    
- 若执行过程中风险升级，必须停止继续扩展，并重新确认任务范围
    
---

# 10. 工作范围授权机制

这是整个规范里最关键的一段。

## 10.1 默认模式：单模块模式

仅允许在一个模块中工作。

例如：

- `platform_auth`
    
- `frontend/src/modules/order`
    

适用于绝大多数任务。

---

## 10.2 跨模块模式：白名单授权

仅当任务明确声明为跨模块开发或联调时，才允许进入父级目录协作。

必须显式声明：

- 主模块
    
- 协作模块
    
- 允许读取范围
    
- 允许修改范围
    
- 禁止修改范围
    

---

## 10.3 系统级模式：少量开放

只在以下情况开放更大范围：

- 前后端全链路联调
    
- 契约结构升级
    
- 错误码体系调整
    
- 登录链路系统级改造
    

即使如此，也必须列白名单，不得默认扫全项目。

---

## 10.4 父级目录原则

进入父级目录不等于可扫描整个项目。

**父级目录只是工作位置，不是无限授权。**

未被明确列入白名单的目录，默认禁止读取或修改。

---

# 11. 跨模块协作规范

## 11.1 跨模块任务必须指定主模块

例如：

- 主模块：`platform_auth`
    
- 协作模块：`business_user`
    
- 契约模块：`contracts`
    

这样可以防止 agent 把跨模块任务理解成系统级重构。

---

## 11.2 跨模块任务的允许范围格式

### 允许读取

- `platform_auth/**`
    
- `business_user/**`
    
- `contracts/**`
    
- 指定 docs 文档
    

### 允许修改

- `platform_auth/src/**`
    
- `business_user/src/**`
    
- `contracts/openapi/**`
    

### 禁止修改

- 未授权模块
    
- 根级配置
    
- 前端或无关仓库
    
- 本任务无关文档
    

---

## 11.3 跨模块停止条件

若任务执行中发现需要新增第 3 个以上后端模块，默认暂停扩展，重新定义任务范围。

避免联调任务膨胀成系统改造。

---

# 12. 前后端联调规范

## 12.1 联调原则

- 联调必须指定主链路
    
- 联调时只开放相关前端模块、后端模块与契约
    
- 不得顺手调整无关页面或后端模块
    

---

## 12.2 常见联调范围示例

### 登录链路联调

允许范围：

- `frontend/src/modules/auth/**`
    
- `platform_auth/**`
    
- `contracts/openapi/auth.*`
    
- `contracts/errors/**`
    

### 下单链路联调

允许范围：

- `frontend/src/modules/order/**`
    
- `business_order/**`
    
- `contracts/openapi/order.*`

# 16. 必须遵守的禁止事项

## 16.1 通用

- 不得为了“优雅”扩大任务范围
    
- 不得用共享目录做垃圾场
    
- 不得隐式跨模块改代码
    
- 不得跳过契约同步
    
- 不得跳过自测步骤
    

## 16.2 前端

- 不得把业务组件过早抽成全局组件
    
- 不得把模块 API 堆进一个全局文件
    
- 不得把所有状态塞进全局 store
    

## 16.3 后端

- 不得让 controller 直接访问数据库
    
- 不得让 ORM 模型直接穿透到 API
    
- 不得跨模块 import 内部实现
    

## 16.4 AI

- 不得默认扫全项目
    
- 不得顺手重构
    
- 不得私自升级依赖
    
- 不得私自改 schema
    
- 不得未授权跨模块协作
