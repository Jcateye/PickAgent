需求 PRD 总览

来源：/Users/haoqi/OnePersonCompany/PickAgent/docs/PRD.md。这里保留产品主线，不把架构建议反向硬套到 PRD 叙事上。

SKU Ready Agent 的核心不是替代 Excel，而是让商品在活动到来之前持续处于 Ready 状态：平时采集和诊断，活动发布后自动模拟准入，并把不确定项送入人工 Review。

目标用户

运营人员、商品管理、供应链、法务/风控、管理层。每类用户围绕“能不能报、为什么不能报、怎么修、谁来确认”形成协同。

MVP 边界

自动采集、自动诊断、自动模拟、自动建议、人工 Review。不真实接入生产后台，不自动改价，不自动提交活动，不替代法务最终判断。

技术方向

TypeScript / Next.js / assistant-ui / Vercel AI SDK / Plasmo / PostgreSQL / Prisma / Zod / 手写 typed workflow。

MVP 使用闭环

01

采集

插件扫描模拟电商后台表格，预览字段质量，用户确认后发送到 /api/ingest。

02

诊断

服务端校验、标准化、入库快照，生成 SKU 健康状态、问题和下一步建议。

03

解析

活动规则文本由 LLM 解析为结构化规则，Zod 校验，低置信度项进入 Review。

04

模拟

规则引擎基于 SKU 最新事实快照计算准入状态：可报名、可修复、需确认、阻塞。

05

Review

不确定、高风险、跨部门确认事项进入结构化 Review 工作台并保留证据链。

PRD 模块 当前设计判断 开发前需要钉死的点
数据源连接器 方向合理，黑客松以 Plasmo + 模拟后台最稳。 拆出 Page Extractor、Platform Adapter、Canonical Normalizer。
SKU 健康档案 业务价值强，但当前缺稳定 SKU 主体。 明确 SkuProfile、最新快照、最新诊断、历史记录的关系。
活动规则解析 LLM + Zod + Review Gate 是正确路线。 LLM 输出必须是有限类型 Canonical Rule DSL。
活动准入模拟 是核心能力，不能写成一次性 demo if/else。 区分 item-level rules 和 set-level rules。
Review 工作台 闭环成立，但对象不能只是文本消息。 ReviewItem 要有 target、reasonCode、assigneeRole、evidenceRefs。
Agent Chat 适合作为自然语言控制台。 Chat 只调用服务层工具，不承载核心业务逻辑。

建议的 Canonical Rule 类型

  • threshold：销量、库存、好评率等阈值。
  • field_compare：活动价不得高于近 30 天最低价。
  • boolean_block：已参加品牌日不可重复报名。
  • data_required：证书编号、材质、克重必填。
  • quota：黄金类目最多 5 个 SKU。
  • manual_review：折扣口径、规则歧义、低置信度输出。
数据实体 职责 MVP 实现建议
SkuProfile 稳定 SKU 主语，支撑“健康档案”。 可用 platform + storeId + skuId 作为 canonical key，先不做复杂跨平台归并。
SkuSnapshot 一次采集事实，保留原始数据和标准化字段。 P0 必须有 rawJson、sourceUrl、rowIndex、collectedAt。
HealthDiagnosis 日常商品健康判断。 不要把活动库存门槛写死成通用健康规则。
ActivityRuleSet 保存活动原文、结构化规则和 manual review items。 LLM 失败时允许使用 fallback demo rules。
SimulationResult 每个 SKU 对某个活动的准入结论。 P0 保存 status、failedRules、repairPlan、evidenceJson。
ReviewItem 结构化人工确认任务。 必须区分数据问题、规则解释、运营决策、合规确认。
WorkflowRun 轻量运行日志。 只记录 ingest/parse/simulate/report 的步骤和错误，不做通用 DAG。

关系 基数 实现约束 为什么需要
Connector -> SkuSnapshot 1:N 每次采集都保留 connectorId、sourceUrl、rowIndex 和 rawJson。 支撑来源追踪、字段质量判断和插件采集失败排查。
SkuProfile -> SkuSnapshot 1:N canonicalKey 建议先用 platform:storeId:skuId，MVP 不做复杂跨平台合并。 避免把一次采集快照误当长期 SKU 档案。
SkuSnapshot -> SkuHealthDiagnosis 1:N 诊断结果必须引用 snapshotId；规则版本变化后可以重新诊断。 保证健康结论能回放到当时的数据事实。
SkuProfile -> CurrentSkuProjection 1:1 ingest 完成后 upsert 最新 snapshot、diagnosis 和 top issues。 让 Dashboard、SKU 详情和 Chat 查询稳定且便宜。
ActivityRuleSet -> ActivitySimulationRun 1:N 同一活动规则可以重复模拟，scopeJson 记录筛选范围和假设条件。 支撑“补货 200 件会怎样”等重跑场景。
ActivitySimulationRun -> ActivitySimulationResult 1:N 每个 result 同时引用 skuProfileId 和 snapshotId。 同时保留长期 SKU 主语和本次模拟使用的数据事实。
ReviewItem -> Source Object N:1 通过 targetType + targetId 统一指向数据问题、规则歧义、模拟结果或高风险动作。 Review 是结构化任务，不是纯文本提醒。
Evidence JSON / EvidenceRef N:N 语义 MVP 先以 evidenceJson 保存 sourceType、sourceId、field、rawValue、normalizedValue、ruleId、collectedAt。 让 Safety Layer 和 Review Gate 变成系统能力，而不是展示文案。

Rule of thumb:
- Client owns interaction, preview, filters, navigation.
- Service owns workflow, permission boundary, validation, and orchestration.
- Domain engine owns deterministic judgement.
- Database stores facts, derived results, review tasks, and audit traces.
客户端入口 API / Route 服务端 Service 核心数据表 边界说明
Plasmo 插件侧边栏 / 模拟后台采集 POST /api/ingest IngestService + Normalizer + HealthAssessmentService Connector, SkuProfile, SkuSnapshot, SkuHealthDiagnosis, CurrentSkuProjection, ReviewItem, WorkflowRun 插件只做 DOM 提取和预览；服务端才做权威校验、标准化、入库和诊断。
Dashboard 总览页 GET /api/health/summary SkuQueryService CurrentSkuProjection, SkuHealthDiagnosis, ReviewItem, WorkflowRun Dashboard 不重新计算健康状态，只读 current projection 和聚合统计。
SKU 健康档案页 GET /api/skus/:skuId SkuQueryService SkuProfile, SkuSnapshot, SkuHealthDiagnosis, ActivitySimulationResult, ReviewItem 页面展示 latest + history，避免把某次 snapshot 当成长期档案。
活动规则解析页 POST /api/activities/parse ActivityRuleService ActivityRuleSet, ReviewItem, WorkflowRun LLM 只输出 Canonical Rule DSL；低置信度、解析失败、歧义规则进入 Review。
活动准入模拟页 POST /api/activities/:id/simulations ActivitySimulationService + RuleEvaluator + ReviewService ActivityRuleSet, CurrentSkuProjection, SkuSnapshot, ActivitySimulationRun, ActivitySimulationResult, ReviewItem 先跑 item-level rules，再跑 set-level rules；活动 eligibilityStatus 不写回日常 healthStatus。
Review 工作台 GET /api/reviews / POST /api/reviews/:id/decision ReviewService + SkuQueryService ReviewItem, SkuProfile, SkuSnapshot, ActivityRuleSet, ActivitySimulationResult Review 是结构化任务；批准、驳回、修改都要记录 decision 和 evidence。
Agent Chat 控制台 POST /api/chat ToolService -> SkuQueryService / ActivitySimulationService / ReportService 按工具调用访问 CurrentSkuProjection, ActivitySimulationResult, ReviewItem, ActivityRuleSet Chat 是自然语言入口，不直接查 DB，不直接做业务判断。
报告页 POST /api/reports ReportService CurrentSkuProjection, ActivityRuleSet, ActivitySimulationResult, ReviewItem 报告只汇总已落库结论和证据，不重新创造业务事实。
外部 Agent 信号卡片 POST /api/agent-events AgentEventService + ActivityRuleService / ReviewService AgentEvent, ActivityRuleSet, ReviewItem A/D Agent 信号作为上下文和触发源，不做复杂 multi-agent event bus。

Prisma 草图的重点是关系方向和 MVP 字段集合，不是最终迁移文件。真实落地时建议先把 enum、Decimal 精度、Json Zod schema、软删除策略和 Review 状态流转再收紧一轮。

具体价值点

价值表达要避开“Excel 也能做”的质疑，重点放在持续准备、证据链和执行闭环。

从临时筛选变成持续 Ready

Excel 解决的是数据准备好以后怎么筛；SKU Ready Agent 解决的是数据如何自动汇聚、商品如何持续健康、活动来临前如何提前修复。

把规则转成可执行检查

平台活动规则、合规规则、战略信号最终都落到 SKU 的库存、价格、证书、活动互斥和 Review 任务上。

LLM 可控使用

模型负责理解和表达，规则引擎负责判断。Zod、证据链、Review Gate 和 fail-closed 策略共同降低幻觉与误判风险。

核心注意点

这些是开发前要确认的边界。确认后再写入 PRD 或架构文档，能减少后续返工。

01. 不要让 Snapshot 冒充档案

SKU 健康档案必须有稳定主语。否则详情页、历史趋势、最新状态、跨来源聚合都会被迫临时拼查询。

02. 日常健康和活动准入要分开

库存 500 是某次活动门槛，不应成为所有 SKU 的长期健康规则。长期 healthStatus 和活动 eligibilityStatus 要分离。

03. 类目上限是集合规则

“黄金类目最多 5 个 SKU”不是逐 SKU 规则，需要先筛候选集合，再按排序策略选择 Top N。

04. healthScore 只能辅助展示

分数不能推翻规则结论。Blocked / Risky 的硬规则必须优先于分数，否则解释成本会很高。

05. Chat 不是核心业务层

Chat 只调用同一套 application service / tool service。页面按钮和聊天工具应复用相同逻辑。

06. Evidence 不能只是字符串

建议至少结构化保存 source、field、rawValue、normalizedValue、ruleId、collectedAt。MVP 可先用 JSON。

建议的 P0 收敛：模拟后台 + 插件或页面采集 + ingest + snapshot + deterministic health + rule parse fallback + simulation + review list。assistant-ui 和报告生成可以做薄，不要阻塞主链路。
