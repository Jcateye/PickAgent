## Context

浏览器插件目前已经有 side panel 与 popup 的静态壳层，但尚未形成真实的业务闭环。现阶段最重要的不是先打通全部后端能力，而是先把插件自身的页面识别、字段提取、采集预览、自动翻页与运行状态做成独立可演示模块，并在模块末尾再接入真实 ingest。

该模块需要遵守两个约束：一是插件只负责采集与采集流程控制，不承担健康诊断、规则解析或 Review 决策；二是开发阶段允许 mock 先行，避免被共享后端能力阻塞。

插件可以展示采集层风险，例如字段缺失、页面结构异常、翻页失败、payload 校验失败或用户需要确认是否继续采集。插件不得展示业务健康结论、活动准入结论、补货建议、合规判断或 Review 决策结果；这些结论必须由服务端和员工/Agent 工作台承接。

## Goals / Non-Goals

**Goals:**
- 让插件能够识别目标商品列表页并判断是否可采集。
- 让插件能够提取当前页字段并生成标准化预览。
- 让插件能够按分页或循环策略获取全量数据并暴露运行状态。
- 让插件在模块最后一步接入真实 ingest API。

**Non-Goals:**
- 不在插件内执行 SKU 健康诊断。
- 不在插件内实现活动规则解析或准入模拟。
- 不在插件内实现 Review、报告或 Agent Copilot 会话。
- 不在本 change 内要求后端先完成全部真实业务逻辑。

## Decisions

1. 采集流程采用“页面识别 → 当前页提取 → 预览确认 → 自动翻页采集 → 最终提交”的固定顺序，而不是允许用户任意跳步。这样可以保证业务模块内部串行，便于分派和验收。
2. 页面识别与字段提取使用可替换的 adapter 结构，但 adapter 只服务于插件内部，不扩展为全系统连接器框架。这样可以支持后续不同页面结构，同时避免过早抽象。
3. 采集运行状态在插件侧建本地 run state，包含当前页、累计记录数、失败原因和人工确认点。这样插件即使在 mock 阶段也能完整演示“自动化获取全部数据”的核心价值。
4. 与服务端的接入采用 contract-first payload。插件侧先固定提交的数据形状，开发期通过 mock adapter 模拟提交，最后再替换为真实 ingest API。
5. 分页自动化优先支持“受控循环”，即每翻一页都保留状态与中断点，而不是一次性黑盒跑完。这样更符合可诊断、可追踪的产品设计约束。
6. 插件文案与 mock 数据必须使用采集层语义，不使用“准入预诊断”“补货建议”“合规判断”等业务结论语义，避免用户误以为插件承担服务端判断。

## Risks / Trade-offs

- [目标页面结构变化频繁] → 通过页面识别规则与字段映射配置分离，降低单一 DOM 结构耦合。
- [自动翻页流程卡住或误翻] → 在插件侧保留运行中断、确认继续和失败回滚提示。
- [过早绑定真实 ingest 契约] → 先冻结最小 payload 结构，只在模块最后做真实接口适配。
- [插件承担过多业务逻辑] → 明确插件只输出采集事实与运行状态，其他业务结论留给后续模块。

## Migration Plan

1. 先用本地 mock 数据与假提交通路完成插件内部闭环。
2. 冻结 payload 结构并与 ingest 接口对齐。
3. 用真实接口替换 mock submit adapter。
4. 在跨模块联调 change 中验证插件到控制台的端到端链路。

## Completion Gate

- 页面识别、当前页提取、预览、受控循环采集、中断/继续和 mock submit 可完整演示。
- 插件 payload schema 已冻结，并能用 fixture 校验字段映射、异常字段和累计记录。
- 插件只输出采集事实、字段映射和采集 run state，不输出健康/准入/Review/报告结论。
- 真实 ingest API 未完成时，本模块可声明“插件侧已完成，不阻塞”；真实提交验收必须等待 `backend-business-foundation` 完成对应 ingest 能力。

## Open Questions

- 插件侧是否需要在提交前允许用户删除异常行。
